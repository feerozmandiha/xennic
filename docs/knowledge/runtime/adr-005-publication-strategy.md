# ADR-005: Publication Strategy

**Status:** Proposed | **Date:** Tir 1405 | **Deciders:** Xennic Architecture Team

---

## Context

Validated knowledge must be published to multiple targets: vector database (Qdrant), knowledge graph (Neo4j), search index (Meilisearch/Elasticsearch), and the GraphQL API layer. Publishing independently to each target risks inconsistency — a document might appear in search but not in the vector DB. Partial failures during publication could leave targets in an inconsistent state, degrading query reliability and user trust.

## Decision

Adopt a versioned all-or-nothing publication strategy:

1. **Prepare** — assemble the publication payload: vector embeddings, graph nodes/edges, search index documents, API response objects
2. **Pre-check** — verify each target is reachable and has capacity; fail fast if any target is unhealthy
3. **Publish** — write to all targets in a coordinated manner using a two-phase commit pattern:
   - Phase 1 (Prepare): each target stages the data but does not make it visible to queries
   - Phase 2 (Commit): after all targets confirm stage success, issue commit to make data visible
   - Rollback: if any target fails commit, all targets revert to pre-publication state
4. **Verify** — confirm data is queryable on each target; rollback automatically if verification fails within a timeout window

Each publication produces a monotonically increasing version number. Versions are tracked per document, enabling point-in-time rollback and audit.

## Alternatives Considered

- **Publish-as-you-go:** Each target receives data independently as it becomes available. Simplest approach but guarantees inconsistency — a document could be in search but missing from the graph for minutes or hours.
- **Sequential per-target:** Publish to target A, then B, then C. If B fails, A is already committed with no rollback mechanism. Partial failure is guaranteed over time at scale.

## Consequences

- **Positive:** Strong consistency across all publication targets. Automated rollback prevents partial-failure states. Versioning enables rollback to any prior state for disaster recovery. Audit trail of every publication event.
- **Negative:** Publication latency increases due to two-phase commit overhead. If a target is slow or unavailable, publication is blocked entirely. Verification step adds additional round-trips.

## Future Impact

The two-phase commit pattern allows adding new publication targets (e.g., a relational data warehouse, a time-series DB for trend data) without modifying existing publication logic. Versioned publication enables future capabilities: A/B testing of knowledge versions, scheduled publication windows, and incremental rollouts.

## Compliance

- Every publication must use the two-phase commit protocol — no direct writes to target stores bypassing the publication service.
- Publication success/failure metrics must be exposed via Prometheus.
- Rollback must be tested in CI/CD pipeline for every target type.
- Version numbers must be generated by the publication service, not by individual targets.
