2. ARCHITECTURE DECISIONS
───────────────────────────────────────────────────────────────────────────────

  ADR-K3.0-001  Event-Driven Pipeline
  ────────────  The factory pipeline is event-driven, not orchestrator-driven.
                Each stage subscribes to the previous stage's events via
                RabbitMQ. An Orchestrator Service is deferred until Phase 2.

  ADR-K3.0-002  EKO as Universal Knowledge Unit
  ────────────  Every piece of engineering knowledge is represented as an
                Engineering Knowledge Object (EKO) with 6 kinds (FACT, RULE,
                CONSTRAINT, ASSUMPTION, CALCULATION, CONCLUSION) and a
                standardized 16-field structure.

  ADR-K3.0-003  CloudEvents Specification
  ────────────  All inter-service events conform to the CloudEvents 1.0
                specification with the `com.xennic.factory.*` type prefix.

  ADR-K3.0-004  Hybrid Search (Vector + Graph)
  ────────────  Retrieval uses Reciprocal Rank Fusion (RRF) to combine dense
                vector search (Qdrant) with knowledge graph traversal, with
                query-type-aware routing.

  ADR-K3.0-005  Factory Services are Stateless
  ────────────  Every factory service is stateless and scales horizontally.
                State is stored in the event stream and the knowledge stores.

  ADR-K3.0-006  Quality Gates Before Publication
  ────────────  Every EKO passes through 5 automated quality gates before
                publication. Scores below 0.3 are rejected; 0.3-0.6 are
                escalated for human review.

  ADR-K3.0-007  Policy Engine from Governance Documents
  ────────────  Quality gates and pipeline policies are derived directly from
                the published governance documents (metadata-schema, taxonomy,
                source-hierarchy, data-quality-policy). No separate policy
                language.

  ADR-K3.0-008  Adapter Pattern for Knowledge Sources
  ────────────  Every future knowledge source implements a common
                `KnowledgeSourceAdapter` interface, enabling pluggable intake
                from any source type.

  ADR-K3.0-009  AI is a Service, Not a Layer
  ────────────  AI capabilities (extraction, classification, enrichment) are
                consumed as services by the factory, not embedded as a layer.
                The AI Service can be replaced without factory changes.

  ADR-K3.0-010  Deployment Progression: Docker → Swarm → K8s
  ────────────  The factory starts on Docker Compose alongside the existing
                stack, graduates to Docker Swarm for Beta, and targets
                Kubernetes for GA.

5. ASSUMPTIONS
───────────────────────────────────────────────────────────────────────────────

  1. The existing NestJS API will remain the synchronous gateway for all
     user-facing operations.

  2. RabbitMQ will be available and stable with at-least-once delivery and
     dead-letter queue support.

  3. Qdrant can scale to the target vector count with acceptable query
     latency. Collection sharding strategy will be validated in Alpha.

  4. LLM providers (API or self-hosted) will be available and maintain
     acceptable extraction quality for engineering content.

  5. The concept registry will be populated iteratively, starting with
     ~100 core concepts and growing to 1,000+.

  6. Human reviewers with engineering domain expertise will be available
     for quality escalation.

  7. The bilingual requirement (FA + EN) will be maintained at every
     pipeline stage — not treated as a translation problem.

  8. Pipeline versioning will follow the strategy defined in
     `runtime/pipeline-versioning.md`.
