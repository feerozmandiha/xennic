================================================================================
3. RUNTIME ARCHITECTURE DECISIONS
================================================================================

RAD-001: Event-Driven Stateful Pipeline
-----------------------------------------
Decision: The runtime is an event-driven pipeline where each stage publishes
events consumed by downstream stages. Document state is tracked in PostgreSQL,
not in-memory.
Rationale: Enables asynchronous processing, graceful degradation, multi-language
support, and distributed deployment. State in DB enables recovery after restart.

RAD-002: 21-Stage Lifecycle
-------------------------------
Decision: 21 explicitly defined stages from document receipt to lifecycle management.
Rationale: Each stage has a single responsibility. Granular enough for detailed
observability and failure recovery. Coarse enough for manageable orchestration.

RAD-003: 15 Internal Services
---------------------------------
Decision: 15 dedicated services, each owning a specific stage or group of stages.
Rationale: Service-per-processor enables independent scaling, deployment, and
failure isolation. No monolithic processing bottleneck.

RAD-004: 8 External Integrations
------------------------------------
Decision: AI Service (soft), Vision Service (soft), Qdrant (hard), Knowledge Graph
(soft), Event Bus (hard), MinIO (hard), PostgreSQL (hard), Knowledge API (soft).
Rationale: Hard dependencies block pipeline; soft dependencies trigger fallback
behavior. Prevents complete pipeline failure when non-critical services are down.

================================================================================
4. PIPELINE DECISIONS
================================================================================

PD-001: Parallel Execution Where Possible
-------------------------------------------
Decision: Concept Resolver ∥ Ontology Resolver, Entity Extractor ∥ Relationship
Extractor, Vector Publisher ∥ Graph Publisher.
Rationale: Reduces end-to-end processing time by ~40% compared to sequential execution.

PD-002: Per-Source-Type Parsers
------------------------------------
Decision: 15 source types with format-specific parsers.
Rationale: A single parser cannot handle PDF, DOCX, HTML, scanned images, and
specialized formats like IEC standards and Tavanir regulations.

PD-003: Human-in-the-Loop Gates
-----------------------------------
Decision: Tier 1-2 sources always reviewed, low-confidence (<0.7) flagged for review,
human review dashboard with 48-hour SLA.
Rationale: Engineering knowledge errors can have safety implications. Automated
extraction cannot achieve 100% accuracy for specialized engineering content.

PD-004: 16-Processor Pipeline
---------------------------------
Decision: 16 processors in defined execution order with parallel branches.
Rationale: Each processor has a single responsibility. The pipeline is composable —
processors can be added, removed, or reordered without changing others.

================================================================================
5. VALIDATION DECISIONS
================================================================================

VD-001: Six-Layer Validation
-------------------------------
Decision: Six independent validation layers applied sequentially: File → Metadata →
Semantic → Engineering → Knowledge → Publication.
Rationale: Each layer catches different error types. Earlier layers are cheaper
and faster (file validation is sub-second). Later layers are more expensive but
more thorough. Fail-fast in early layers saves processing resources.

VD-002: Tier-Dependent Approval
-----------------------------------
Decision: Tier 1-2 sources require human review for Engineering Validation.
Tier 3-5 can auto-pass Engineering Validation.
Rationale: Engineering standards and regulations have higher authority and lower
error tolerance. Manufacturer data and community knowledge can be auto-validated.

VD-003: Quality Gates with Pass/Fail/Flag
---------------------------------------------
Decision: Three outcomes per validation check: PASS (proceed), FAIL (reject),
FLAG (human review).
Rationale: Not all failures are equal. Some are permanent (corrupt file → reject),
some are transient (low OCR confidence → flag for human review).

================================================================================
6. EVENT ARCHITECTURE DECISIONS
================================================================================

EAD-001: 30 Events in 6 Categories
--------------------------------------
Decision: 30 distinct event types across ingestion, processing, validation,
publication, lifecycle, and system categories.
Rationale: Granular enough for precise orchestration and observability. Each event
triggers a specific action in a specific subscriber.

EAD-002: Standard Event Envelope
------------------------------------
Decision: Every event has: id (UUID), type, version, timestamp, source,
correlation_id, causation_id.
Rationale: correlation_id enables per-document tracing across all events.
causation_id enables causality chain reconstruction. Version field enables
event schema evolution.

EAD-003: At-Least-Once Delivery with Idempotent Consumers
------------------------------------------------------------
Decision: RabbitMQ at-least-once delivery. Consumers deduplicate by event.id
with 7-day window.
Rationale: Exactly-once delivery is expensive and not needed. Idempotent consumers
ensure duplicates don't cause side effects.

7. DEPENDENCY DECISIONS
================================================================================

DD-001: Hard vs Soft Dependency Classification
-------------------------------------------------
Decision: Hard dependencies: Event Bus, Object Storage, PostgreSQL, Qdrant,
Embedding Service. Soft dependencies: AI Service, Vision Service, Knowledge
Graph, Knowledge API.
Rationale: Hard dependencies cannot be bypassed; the pipeline stops without them.
Soft dependencies have fallback strategies (pattern-based extraction, deferred
graph updates).

DD-002: Critical Path Identified
------------------------------------
Decision: The minimum processing path is: Upload → Parser → Cleaner → Metadata →
Concept → Semantic → Entity → Chunk → Embedding → Vector → Publication (30-120s).
Rationale: This path excludes OCR (scanned documents) and human review (flag only).
Non-critical paths can be deferred without blocking publication.

================================================================================
8. PERFORMANCE DECISIONS
================================================================================

PF-001: Performance Targets by Operation
-------------------------------------------
Decision: Specific P50/P95/P99 targets defined for all operations (file validation:
100ms/500ms/1s, embedding 100 chunks: 5s/15s/30s, end-to-end text: 30s/120s/300s).
Rationale: Targets enable performance regression detection. Deviations trigger
alerts and investigation.

PF-002: GPU Strategy: Batch Embedding
-----------------------------------------
Decision: GPU used exclusively for embedding generation. Batches of 32-64 chunks
for GPU efficiency. CPU fallback if GPU unavailable.
Rationale: Embedding is the only GPU-intensive operation. Batching maximizes GPU
utilization. CPU fallback ensures pipeline continuity.

PF-003: Lazy Embedding for Low-Priority Documents
-----------------------------------------------------
Decision: High-priority documents embedded immediately. Low-priority documents
queued for off-peak batch embedding. On-demand embedding for first query.
Rationale: Optimizes GPU resource usage. Most low-priority documents may never
be queried; avoid embedding them unnecessarily.

================================================================================
9. SCALABILITY DECISIONS
================================================================================

SCD-001: Stateless Processors with Auto-Scaling
---------------------------------------------------
Decision: All processors are stateless. State stored in PostgreSQL and Event Bus.
Worker pools auto-scale based on queue depth (target <1000).
Rationale: Stateless services scale horizontally without coordination overhead.
Queue-depth-based scaling prevents backlog and ensures responsive processing.

SCD-002: Multi-Tenant Isolation
-----------------------------------
Decision: Per-workspace Qdrant collections, graph namespaces, storage quotas.
Rationale: Prevents cross-workspace data leakage. Enables fair resource allocation.
Supports workspace-level billing and usage tracking.

SCD-003: Scaling Limits Defined
-----------------------------------
Decision: 100K docs/day target, 500K burst, 100MB max per document, 10TB per
workspace, 100B vectors total (theoretical).
Rationale: Provides clear capacity planning targets. Burst capacity handles
batch imports and crawler dumps.

================================================================================
10. AI INTEGRATION DECISIONS
================================================================================

AID-001: AI as Soft Dependency for Extraction
-------------------------------------------------
Decision: Concept/entity/relationship extraction uses AI Service by default.
Falls back to pattern-based extraction (regex, NER) when AI unavailable.
Rationale: AI provides higher quality extraction. Pattern-based fallback ensures
pipeline continues during AI Service outages.

AID-002: AI as Hard Dependency for Embedding
------------------------------------------------
Decision: Embedding generation requires AI Service. No fallback (publication
blocked without embeddings).
Rationale: Embedding is a hard requirement for RAG retrieval. The knowledge base
is useless without vector search capability.

AID-003: AI-Based Quality Scoring
-------------------------------------
Decision: AI Service computes confidence scores, hallucination risk, and
validation scores per confidence-scoring.md.
Rationale: Automated quality scoring at scale. Reduces human review workload
by auto-accepting high-confidence documents.

12. RISKS AND FUTURE CONSIDERATIONS
================================================================================

RISK-01: Pipeline Complexity
------------------------------
25 components (15 services + 8 integrations + event bus + orchestrator) is
a complex distributed system. The failure modes are numerous.
Mitigation: Comprehensive observability (RAD-008), structured failure recovery
(ADR-007), and explicit dependency map (runtime-dependency-map.md).

RISK-02: OCR Quality for Persian Documents
---------------------------------------------
Persian OCR accuracy is lower than English, especially for mixed RTL/LTR
engineering documents with tables and formulas.
Mitigation: OCR confidence threshold (0.7), human review for low-confidence
scans, Vision Service integration for specialized Persian OCR pipeline.

RISK-03: Pipeline Performance at Scale
-----------------------------------------
100K documents/day with full processing (OCR, extraction, embedding) requires
significant compute resources. Batch imports could create processing spikes.
Mitigation: Auto-scaling worker pools, priority queues, rate limiting, lazy
embedding, batch embedding GPU optimization.

RISK-04: Eventual Consistency Challenges
-------------------------------------------
Vector DB and Knowledge Graph are updated asynchronously. There is a window
where a document is vector-published but not graph-published.
Mitigation: Publication Coordinator ensures all-or-nothing. In-flight
documents tagged to prevent incomplete retrieval.

RISK-05: Human Review Bottleneck
------------------------------------
Tier 1-2 documents always require human review. If ingestion rate exceeds
review capacity, the pipeline backs up.
Mitigation: Reviewer pool scaling, 48-hour SLA with escalation, priority
queuing for critical documents, confidence-based auto-approval for Tier 3-5.

RISK-06: Pipeline Version Migration
--------------------------------------
When the pipeline changes (new model, new processor), existing documents
may need reprocessing. Full reprocessing is expensive.
Mitigation: Staged migration by priority, incremental re-embedding, backward
compatibility for MINOR versions, selective reprocessing for active documents.

13. OPEN ARCHITECTURAL QUESTIONS
================================================================================

OAQ-001: Knowledge Graph Technology Choice
----------------------------------------------
Neo4j vs Apache Age (PostgreSQL extension) vs Amazon Neptune. The runtime
architecture is graph-technology-agnostic. Final choice depends on deployment
environment (self-hosted vs cloud) and team expertise.

OAQ-002: Embedding Model Lifecycle
--------------------------------------
How often should embedding models be updated? What is the validation process
for new models? The architecture supports model versioning, but the governance
for model updates is not defined.

OAQ-003: Multi-Region Processing
------------------------------------
The architecture supports geographic distribution, but the data sovereignty
requirements (where can documents from country X be processed and stored?)
are not fully defined.

OAQ-004: Connector SDK Scope
--------------------------------
The future connector framework is mentioned but not designed. Should it support
bi-directional sync (pull from SharePoint AND push updates back)? What is the
minimum viable connector API?

OAQ-005: Cost Model for Processing
-------------------------------------
Each document consumes compute (CPU for parsing, GPU for embedding, human time
for review). How are these costs tracked and attributed to workspaces?
