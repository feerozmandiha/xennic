3. ARCHITECTURE DECISIONS
================================================================================

AD-001: Knowledge Base as the AI Source of Truth
-------------------------------------------------
Decision: The knowledge base (docs/knowledge/) is designated as the PRIMARY
source of truth for all engineering AI responses. No AI-generated engineering
answer may be presented without source evidence from this knowledge base.
Rationale: Prevents hallucination, ensures traceability, and establishes
Xennic as an authoritative engineering platform rather than a generic LLM
chatbot.

AD-002: Layered Governance Structure
-------------------------------------
Decision: Governance is split into 6 complementary documents (metadata-schema,
taxonomy, ontology, naming-conventions, data-quality-policy, source-hierarchy)
rather than one monolithic document.
Rationale: Each document addresses a distinct concern. Metadata defines WHAT
to store, taxonomy defines HOW to classify, ontology defines HOW entities
relate, naming defines HOW to reference, quality defines HOW to validate,
and hierarchy defines HOW to trust.

AD-003: 5-Category Metadata Architecture
-----------------------------------------
Decision: Metadata is decomposed into 5 orthogonal categories:
  A. Core — universal fields for all document types
  B. Electrical Engineering — domain-specific technical parameters
  C. AI Intelligence — confidence, reasoning, verification metadata
  D. RAG — retrieval, chunking, embedding parameters
  E. Traceability — provenance, source chain, supersession
Rationale: Separation allows independent evolution of each category. Core
metadata is stable across all document types; Electrical metadata is specific
to engineering domain; AI/RAG/Traceability metadata support the AI
intelligence layer without polluting base document metadata.

AD-004: Source Tier System (Tier 1-5)
--------------------------------------
Decision: Engineering sources are ranked in a 5-tier hierarchy:
  Tier 1: International Standards (IEC, IEEE, ISO)
  Tier 2: National Regulations (ISIRI, Tavanir)
  Tier 3: Manufacturer Documentation
  Tier 4: Peer Reviewed Papers
  Tier 5: Community Knowledge
Rationale: This hierarchy enables automated authority-based filtering during
RAG retrieval. AI can prioritize higher-tier sources and flag when only
low-tier sources exist. It also enables the "No Tier N citation when Tier
N-1 exists" rule.

AD-005: 4-Mode Reasoning Framework
-----------------------------------
Decision: The AI employs 4 reasoning modes depending on query type:
  - Deductive: Standards application, code compliance
  - Inductive: Fault diagnosis, pattern recognition
  - Abductive: Troubleshooting, root cause analysis
  - Case-Based: Design assistance, equipment selection
Rationale: Engineering problems require different reasoning approaches.
A single reasoning mode cannot cover all use cases. The framework allows
the AI to select the appropriate mode and document which mode was used.

AD-006: Mandatory Response Structure
--------------------------------------
Decision: Every AI response must contain 5 sections:
  1. Answer (direct response)
  2. Reasoning Path (step-by-step derivation)
  3. Evidence (sources with tiers)
  4. Confidence (numerical score)
  5. Limitations (what the answer does NOT address)
Rationale: Forces transparency. Users can verify every claim. The reasoning
path enables audit of AI decision-making. Limitations prevent over-reach.

AD-007: Weighted Confidence Scoring
-------------------------------------
Decision: Confidence = 0.30×Source + 0.25×Retrieval + 0.25×Reasoning
                    + 0.10×Temporal + 0.10×Consensus
Rationale: Source authority and retrieval relevance are the most important
factors (combined 55%). Reasoning adds another 25%. Temporal recency and
multi-source consensus are supporting factors. The formula is documented
and auditable.

AD-008: Hallucination Prevention as Multi-Layer Defense
---------------------------------------------------------
Decision: 5 prevention layers applied sequentially:
  1. Knowledge Boundary Detection (can the KB answer this?)
  2. Source Verification (every claim mapped to source chunk)
  3. Constraint Enforcement (no numbers/designs/safety without source)
  4. Confidence Threshold Gates (flags at <0.5 and <0.3)
  5. Human Escalation (safety-critical, conflicting Tier 1, low confidence)
Rationale: No single technique prevents all hallucination. Defense-in-depth
with escalating strictness catches failures at each layer.

7. RISKS AND FUTURE CONSIDERATIONS
================================================================================

RISK-01: Standards Copyright
----------------------------
Most IEC and IEEE standards are paywalled. Xennic cannot legally ingest
full-text standards without licensing. Mitigation: Focus on publicly
available abstracts, summaries, and open standards. For paywalled content,
provide citation metadata and purchase links rather than full text.

RISK-02: Persian Language NLP Quality
-------------------------------------
Persian embedding models may not match English model quality. The taxonomy
and ontology assume bilingual capability. Mitigation: Evaluate multilingual
embedding models (e.g., paraphrase-multilingual-MiniLM-L12-v2) and consider
hybrid Persian-English retrieval.

RISK-03: Source Hierarchy Enforcement Complexity
-------------------------------------------------
Automated enforcement of "no Tier N when Tier N-1 exists" requires
sophisticated query understanding and source mapping. This may be difficult
to implement correctly. Mitigation: Implement as a soft warning initially,
then hard enforcement after validation.

RISK-04: Maintenance Burden of Governance Docs
-----------------------------------------------
The 6 governance documents must stay synchronized with implementation.
If the API changes field names or taxonomy structure, governance docs must
be updated. Mitigation: Reference governance docs in CI/CD review process.
Add links in Prisma model comments to governance document sections.

RISK-05: Confidence Scoring Calibration
----------------------------------------
The weighted formula (0.30/0.25/0.25/0.10/0.10) is a starting point. It
may not reflect actual answer quality across all engineering domains.
Mitigation: Plan calibration sprint after initial deployment; collect user
feedback on confidence accuracy; adjust weights based on real data.

RISK-06: Empty Directories
---------------------------
5 directories (standards, tariffs, manufacturers, catalogs, rag) are
currently empty. Future sprints must populate these or remove them to
avoid confusion. Mitigation: Populate standards/ and rag/ in Sprint K2-K3;
tariffs/, manufacturers/, catalogs/ are medium-term (K4).
